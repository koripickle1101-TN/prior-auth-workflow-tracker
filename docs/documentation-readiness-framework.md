# Documentation readiness framework

Prior authorization workflow quality depends on whether the request is complete before payer review.

## Required readiness checks

Clinical note

The note should support the requested service and explain medical necessity in plain operational terms.

Order details

The order should match the requested service, body site, frequency, duration, and provider direction.

Diagnosis support

The diagnosis should match payer requirements for the requested service.

Service code check

The service code should match the request type and payer category.

Payer-specific requirement

The request should be checked against payer-specific forms, portals, attachments, and follow-up timing.

## Readiness statuses

Ready

All required documentation is present and the request can move forward.

Needs review

The request has enough information to begin review, but one item needs confirmation.

Incomplete

The request should not be submitted until the missing item is corrected.

Blocked

The request cannot move forward because a required clinical, payer, or ownership element is missing.

## Operational defect sources

Documentation defects may come from:

- incomplete provider notes
- missing medical necessity support
- wrong service details
- unclear diagnosis support
- payer form mismatch
- missing attachment
- missing owner for follow-up

## Portfolio purpose

This framework shows how documentation readiness can be tracked as an operational control point before authorization delay turns into patient access risk or denial risk.
